# File Splitting Rule

## 目的
- 変更容易性を上げつつ、要求を満たす最小変更を維持する。

## 禁止事項
- 無関係なリファクタリングを行わない。
- 分割に便乗した設計変更を行わない。
- 分割に便乗した最適化を行わない。
- 分割に便乗した命名一括変更を行わない。
- 分割のためだけに依存関係を追加しない。

## 分割の発動条件（目安）
- ファイルが大きく、変更時の見落としや衝突が起きやすい。
- 関数が大きく、複数責務が混在している。
- 重複実装があり、修正漏れリスクがある。
- 分岐追加が増え、条件漏れが起きやすい。
- 行数や分岐数は目安として扱う。例として、ファイル行数は 400 から 800 行程度、関数行数は 80 から 150 行程度、主要分岐は 5 から 10 件程度を超えると分割を検討するが、いずれも必須条件ではない。

## 分割単位
- 一つのモジュール（ファイル）は一つの概念に寄せる。
- 公開インターフェースは最小にし、外部公開する関数や型を必要以上に増やさない。
- 表示と入力、状態更新、外部副作用を混在させない。
- 外部副作用には、ファイル入出力、ネットワークアクセス、オペレーティングシステム操作、プロセス起動などを含める。

## 依存方向
- 依存方向は 入出力層 -> 中核ロジック層 -> 副作用層 の一方向とする。
- 逆方向参照を禁止する。
- 共有する型は中核ロジック層に置く。

## 分割手順
1. 新規ファイルを作成する。
2. 既存の関数や構造を中身を変えずに丸ごと移動する。コピペ移動のみとし、ロジック変更、引数変更、型変更、整形、再設計を同時に行わない。
3. 呼び出し側の参照だけを修正する。
4. ビルドまたは最小確認を実行する。
5. 必要な場合のみ整形し、挙動を変えない。
6. 整形や引数整理が必要な場合は、手順4の成功後に別ステップ（別作業）として分離して実施する。

## 完了条件
- ビルドまたは最小確認が成功している。
- 分割理由を1文から2文で記録している。記録には、何が危険だったかと、何を避けたかの2要素を必ず含める。
